A3 (Three Layer Abstraction) Programming Framework

 

 

Majority of the business code, irrespective of programming language, written today has three stages

         Interpret optional inbound request

         Process inbound request data through business logic. For processing/persistence, it refer to other dependent libraries, services, classes, etc

         Return optional outbound response

 

 

 

 

However, porting code from one platform to another becomes a mammoth challenge as all three stages in some form or the other, are tightly coupled.

 

*For example, business code using on-prem persistence storage, when ported to cloud and want to use cloud native persistence storage, it require changes in your business code.

A3 programming framework is available in java programming language supporting JRE version 1.5 or later

Implementation - A3 Programming Framework In Java

 

 

With A3 programming paradigm, all three stages (as defined below) are abstracted from business code, making code portability across platform a smooth experience.

         Layer 1 Abstraction From inbound Request format to business code payload + business code output to outbound response format

         Layer 2 Abstraction Dependent services and libraries referred by business code

         Layer 3 Abstraction Instantiating dependent services and libraries referred by business code

Solution - A3 (Three Layer Abstraction) Programming Framework

 
 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

 

Using java based A3 programming paradigm framework, we have built A3 programming framework for AWS cloud with following implementation

         Layer 1 Abstraction

o   All supported AWS Lambda triggered event type such as API Gateway, SNS, SQS, DynamoDb Stream, etc to A3 payload

o   A3 response in json format

         Layer 2 Abstraction

o   Majority of commonly used AWS service i.e. AWS SNS, SQS, S3, DynamodDB, SES, Kinesis, Secret Manager and State Machine (step functions)

         Layer 3 Abstraction

o   Instantiating AWS services listed in Layer 2 Abstraction using AWS Resource ARN.

         CI/CD Pipeline

o   Pipeline to deploy dependent services/jar/libs as AWS Layer

o   Pipeline to deploy AWS Lambda

o   Reference Cloudformation template (Infra-as-code) for provisioning AWS services listed in Layer 2 abstraction

 

Benefits

         Developer can start work on cloud native development using AWS Lambda without having AWS cloud knowledge

         Developers need to focus only on developing business code, rest all plumbing from intercepting request, transformation, creating and injecting cloud services in business code, etc is taken care by framework

         Business code remains cloud-agnostic and could be migrated across cloud providers such as AWS, Azure, etc with minimal or no changes.

         Flexibility to write custom implementation across abstraction layer. For example, writing layer 2 custom implementation for on-prem mongo db
